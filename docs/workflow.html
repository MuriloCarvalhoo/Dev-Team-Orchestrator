2

/dev-team-start — orquestracao inicial (10 etapas)

Comando sequencial que prepara tudo antes do ciclo autonomo. Cada sub-etapa e executada em ordem; a unica pausa e o gate de aprovacao de wireframes.

  1. Cria estrutura de board/, docs/, tests/ e inicializa DECISIONS.md + PROGRESS.md.
  2. PO gera wireframes em docs/wireframes/{nome}.html — HTML puro sem CSS, cobrindo todas as funcionalidades do PRD, com botoes/inputs nomeados e estados (loading/erro/vazio/sucesso) declarados.
  3. Tech Lead revisa cobertura e consistencia dos wireframes contra o PRD.
  4. Gate do usuario: execucao pausa, usuario aprova manualmente o docs/wireframes/index.html. Feedback reabre o ciclo (max 3 iteracoes). Se rejeitado na 3a, Tech Lead registra DEC-TL-XXX com as ressalvas e decide: abortar ou prosseguir com escopo reduzido.
  5. Tech Lead define stack com versoes exatas (backend, frontend, banco, runners de teste) e registra em DECISIONS.md.
  6. Tech Lead gera contratos em docs/contracts/{nome}.md — UM arquivo por feature com secoes ## API (endpoints, schemas, erros) e ## Screen (componentes, eventos, estados, validacoes). Schemas sao concretos em TypeScript inline. Decisoes nao-triviais (auth, paginacao, status codes) viram DEC-TL-XXX em DECISIONS.md.
  7. PO cria tarefas em board/todo/ referenciando contratos pelo slug (ex: BACK-003: login aponta para docs/contracts/login.md).
  8. DevOps faz scaffold: configs, dependencias, scripts, Playwright setup, Docker — garante que o projeto roda sem erros.
  9. PO injeta contexto tecnico em cada tarefa (paths, dependencias, referencias cruzadas).
  10. Board pronto — usuario pode rodar /dev-team-run.
Reconciliacao wireframe ↔ stack: wireframes sao agnosticos de stack. Se a stack escolhida na etapa 5 invalidar uma suposicao implicita do wireframe (ex: wireframe assume push em tempo real mas stack nao tem WebSocket), Tech Lead abre mini-loop de wireframe (volta para etapa 2) com um DEC-TL-XXX justificando.

3

/dev-team-run — loop autonomo

Autonomo: lanca tarefas independentes em paralelo usando worktrees isolados. A cada iteracao, board-scanner resolve dependencias (depends_on) e ordena as tarefas prontas por priority DESC, created ASC. Loop continuo ate tudo em verified/.

Circuit breaker: cada tarefa original tem orcamento de ate 3 ciclos de FIX-*. Na 4a falha, a tarefa vai para blocked/ com flag needs_user e o loop continua com as demais. Tech Lead tenta desbloquear; se nao conseguir, escala para o usuario.
Backend Frontend DevOps
4

Implementacao + Testes (Dev)

Cada agente dev le sua tarefa (board/todo/{ID}.md) e o contrato apontado no Context, move para in_progress/, implementa o codigo seguindo o contrato (schemas e eventos sao fonte de verdade), escreve e roda testes unitarios e de integracao e move para done/ ao concluir. Decisoes tecnicas novas sao registradas em DECISIONS.md.

Falha persistente de teste ou ambiguidade de contrato: se o agente nao resolve apos tentativas razoaveis, move para blocked/ com reason preenchido no Handoff. Tech Lead assume; se nao conseguir, escala para usuario.
Backend Frontend
5

QA com Playwright E2E

QA Agent valida cada tarefa em done/: re-executa unit/integration (do Dev), escreve specs E2E com Playwright (1 spec por criterio de aceite, getByRole/getByLabel, sem waitForTimeout), captura screenshots por criterio. Screenshots sao a fonte de verdade.

Bug critico = tarefa nao passa: (a) AC sem screenshot correspondente, (b) E2E vermelho, (c) screenshot nao confirma o AC, (d) unit/integration quebrando em re-run. Qualquer um → QA cria tarefa nova FIX-{ID}-N em board/todo/ apontando para a tarefa original; a original vai para verified/ com flag has_fix. Nao se move a tarefa original de volta.

Paralelismo & Merge

Como /dev-team-run coordena varias tarefas em paralelo sem corromper o board nem perder trabalho.

Ciclo de vida do worktree

  1. /dev-team-run cria um git worktree por tarefa paralela, em branch task/{ID} a partir de main.
  2. Dev agent trabalha isolado: implementa, roda unit/integration, move o arquivo via git mv dentro do worktree.
  3. Ao mover para done/, agent faz git fetch + git rebase main, resolve conflitos e faz merge squash no main.
  4. Worktree e removido apos merge. QA opera no main, nunca em worktree.

Board vive no main

O board (todo/, in_progress/, done/, verified/, blocked/) e fonte unica e mora sempre no branch main. Mudancas dentro de um worktree so ficam visiveis depois do merge.

/dev-team-status e board-scanner sempre leem o main, nunca agregam worktrees ativos.

Arquivos append-only (merge automatico)

Os arquivos abaixo sao escritos por multiplos agentes em paralelo e usam driver de merge union configurado em .gitattributes, o que junta linhas sem conflito:

  • docs/DECISIONS.md
  • docs/PROGRESS.md

Entradas em DECISIONS.md sao sempre anexadas ao fim com DEC-{AGENT}-{NNN} unico.

Arquivos estruturais (serializacao)

Exigem rebase + resolucao manual:

  • package.json, tsconfig.json, vite.config.ts (configs)
  • docs/contracts/*.md (fonte de verdade de API/Screen)
  • Arquivos de codigo em paths compartilhados

Conflito nao resolvivel pelo agent → tarefa vai para blocked/ com reason: merge_conflict. Tech Lead assume.

Contratos sao imutaveis durante o run

Apos /dev-team-start, os contratos em docs/contracts/ sao congelados. Se um dev descobrir que precisa alterar um contrato, a tarefa vai para blocked/ com reason: contract_change. Tech Lead atualiza o contrato, registra DEC-TL-XXX e todas as tarefas dependentes sao re-validadas.

Limite de paralelismo

/dev-team-run respeita um teto configuravel (default: 3 worktrees simultaneos) para evitar explosao de conflitos e exaustao de recursos (dev servers, portas, DB schemas para Playwright).

Artefatos de Design

docs/wireframes/

Um arquivo HTML por tela, sem CSS, com botoes/inputs nomeados e estados (loading / erro / vazio / sucesso) declarados. Aprovados pelo usuario antes de qualquer tarefa ser criada.

Escopo da aprovacao: validam estrutura, elementos e estados, nao estetica/layout visual. Estetica e implementada pelo frontend na etapa de Dev (Step 4) e validada por screenshots do QA.

docs/contracts/

Um {nome}.md por feature com duas secoes fixas:

  • ## API — endpoints, request/response schemas, codigos de erro.
  • ## Screen — componentes, eventos, estados, validacoes.

Formato de schema: TypeScript interface/type inline, acompanhados de um exemplo JSON de request e response. Sem OpenAPI, sem JSON Schema — padronizado para evitar drift entre features.

Backend e frontend implementam contra o mesmo arquivo, referenciado pelo slug no titulo da tarefa (ex: BACK-003: login → docs/contracts/login.md). Contratos sao imutaveis apos /dev-team-start (ver "Paralelismo & Merge").

docs/DECISIONS.md

Registro append-only de decisoes tecnicas no formato DEC-{AGENT}-{NNN}. Cada entrada tem: titulo, contexto, opcoes consideradas, decisao, consequencia.

Fronteira vs contratos:

  • Contratos: o que a interface expoe (schema do endpoint de login).
  • DECISIONS: por que aquela escolha (por que JWT em vez de sessao, por que paginacao por cursor).

Se uma decisao muda o contrato, ela aparece nos dois: entrada em DECISIONS + atualizacao do contrato (via blocked/ → Tech Lead).

Agentes

PO Agent

Product Owner
Analisa PRDs, cria tarefas atomicas com criterios de aceite testaveis, mapeia dependencias, injeta contexto tecnico.
Tools: Read Write Edit Glob

Tech Lead

Arquiteto
Define stack com versoes exatas, resolve conflitos tecnicos, desbloqueia tarefas BLOCKED, registra decisoes em DECISIONS.md.
Tools: Read Write Edit Glob Grep

Backend Agent

Desenvolvedor Backend
Implementa tarefas [BACK]: APIs, logica de negocio, banco de dados, integracoes. Escreve testes unitarios e de integracao.
Tools: Read Write Edit Bash Glob Grep

Frontend Agent

Desenvolvedor Frontend
Implementa tarefas [FRONT]: componentes, telas, integracao com APIs. Sempre trata 3 estados: loading, error, empty.
Tools: Read Write Edit Bash Glob Grep

QA Agent

QA Engineer
Valida tarefas DONE com 3 camadas: unit, integration, E2E Playwright. Screenshots sao prova de aceite. Cria tarefas FIX-* para bugs.
Tools: Read Write Edit Bash Glob Grep

DevOps Agent

Infraestrutura
Scaffold do projeto: configs, dependencias, scripts, Playwright setup, Docker. Garante que o projeto roda sem erros.
Tools: Read Write Edit Bash Glob Grep
Skills

task-reader

Le e parseia arquivo de tarefa board/{status}/{ID}.md. Extrai frontmatter, descricao, criterios, contexto e handoff.

task-writer

Atualiza tarefa e move entre pastas de status via git mv. Preenche Log, Handoff e PROGRESS.md.

board-scanner

Escaneia frontmatter de todos os arquivos do board. Resolve dependencias e ordena tarefas prontas.

Piramide de Testes
E2E Playwright + Screenshots
Escrito e rodado pelo QA Agent — fonte de verdade para aceite
Integration Tests
Escrito e rodado pelo Dev Agent, re-executado pelo QA antes do E2E
Unit Tests
Escrito e rodado pelo Dev Agent, re-executado pelo QA antes do E2E
Isolamento do Playwright em paralelo: cada QA worker sobe o stack alvo em porta aleatoria e usa schema de banco isolado (ou DB em memoria). Fixtures e storage-state sao por worker. Sem waitForTimeout, sempre expect().toBeVisible()/toHaveText() para estabilizar.
