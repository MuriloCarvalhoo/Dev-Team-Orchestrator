Workflow Brownfield

Adocao de repositorios com codigo existente

← Voltar para o inicio Ver workflow greenfield →
Proposta — nao implementado Este documento descreve um fluxo pretendido para rodar o Dev Team Orchestrator em repositorios que ja contem codigo em producao. O comando /dev-team-brownfield e o agente code-archaeologist citados aqui ainda nao existem — serao implementados em entregas seguintes (ver Roadmap no final). O workflow greenfield em workflow.html permanece como esta.
Greenfield vs Brownfield
Aspecto Greenfield (/dev-team-start) Brownfield (/dev-team-brownfield)
Ponto de partida Repo vazio ou esqueleto. PRD descreve o sistema do zero. Repo em producao. PRD descreve features novas, extensoes ou refactors.
Stack Tech Lead decide stack do zero e registra em DECISIONS.md. Tech Lead detecta stack via snapshot do code-archaeologist e documenta o que ja existe.
Scaffold DevOps cria package.json, configs, instala deps, sobe Playwright. DevOps em modo "validar, nao criar": confere infra existente, instala apenas o gap (ex: Playwright).
Wireframes Para todas as telas do PRD. Apenas para UI nova ou alterada. Telas existentes nao sao retro-documentadas.
Contratos Tudo new: — contratos descrevem APIs/telas a serem criadas. Diferenciam extends: (referenciam codigo existente por path:line) de new:.
Tarefas BACK, FRONT, DEVOPS. Mesmos tipos + classificacao: FEATURE, EXTEND, REFACTOR.
Risco principal Baixo — nada existe ainda. Alto — sobrescrever/quebrar codigo legado. Recomendado rodar em branch isolada.
Fluxo Brownfield
0

NOVOCode Archaeologist faz varredura do repo

Antes de qualquer outra coisa, o agente code-archaeologist escaneia o repositorio existente para entender com o que esta lidando. Output: um snapshot estruturado que alimenta todos os passos seguintes.

  1. Detecta linguagem e stack lendo manifests: package.json, go.mod, pyproject.toml, Cargo.toml, pom.xml, Gemfile, composer.json.
  2. Detecta framework (backend e frontend) via configs: tsconfig.json, vite.config.ts, next.config.js, nest-cli.json, django settings, etc.
  3. Mapeia rotas/endpoints existentes via Grep nos paths convencionais (routes/, controllers/, handlers/, api/) com referencia file:line.
  4. Detecta runners de teste ja configurados (Jest, Vitest, pytest, go test) e se Playwright ja esta instalado.
  5. Lista scripts disponiveis (npm run, Makefile targets, etc.) e infra (Dockerfile, docker-compose, CI configs).
  6. Gera docs/repo-snapshot.md — relatorio estruturado consumido pelo Tech Lead e PO nos passos seguintes.
Read-only: nesta fase o code-archaeologist nao escreve nada alem de docs/repo-snapshot.md. Nenhuma modificacao em codigo, configs ou dependencias.
Code Archaeologist
1

Tech Lead documenta a stack existente

Tech Lead nao decide stack — le docs/repo-snapshot.md e popula a tabela Stack em DECISIONS.md com o que ja existe. Cada item vira uma decisao DEC-TL-ADOPT-XXX registrando "stack adotada do repo, nao escolhida".

Gap de E2E: se o snapshot indicar que Playwright nao esta instalado, Tech Lead registra DEC-TL-ADOPT-XXX propondo adicionar Playwright como E2E (DevOps instala depois no PASSO 6, sem mexer em testes existentes).
Tech Lead
2

PO classifica o PRD contra o repo

PO le PRD + docs/repo-snapshot.md e classifica cada requisito como:

  1. FEATURE — funcionalidade totalmente nova, sem codigo equivalente.
  2. EXTEND — estende um endpoint/tela ja existente (referencia o path:line do snapshot).
  3. REFACTOR — substitui codigo existente sem mudar a interface externa.
PO Agent
3

PO cria wireframes apenas para UI nova/alterada

Mesma mecanica do greenfield (HTML puro, estados nomeados) mas com escopo restrito. Telas existentes que nao serao tocadas nao ganham wireframe — permanecem como estao.

EXTEND com mudanca de UI: se um EXTEND altera uma tela existente, o wireframe descreve apenas o delta (campos novos, novos estados), nao a tela inteira.
PO Agent
4

Gate de aprovacao do usuario

Identico ao greenfield: usuario aprova docs/wireframes/index.html. Limite de 3 iteracoes. Se rejeitado na 3a, Tech Lead registra DEC-TL-ADOPT-XXX e decide entre abortar ou prosseguir com escopo reduzido.

Usuario
5

Tech Lead cria contratos com extends: vs new:

Mesmo formato do greenfield (## API + ## Screen), mas cada contrato declara explicitamente sua relacao com o codigo existente:

  1. new: — contrato cria endpoint/tela do zero. Schema completo, sem referencias a codigo legado.
  2. extends: src/api/users.ts:42 — contrato adiciona campos ou novos verbos a um endpoint existente. O contrato descreve somente o delta, e referencia o arquivo/linha onde o codigo atual vive.
  3. refactors: src/legacy/auth.js — contrato mantem a interface externa identica e descreve apenas a nova implementacao interna.
Verificacao: antes de salvar um contrato com extends:, Tech Lead confere via Grep que o path:line referenciado realmente existe no repo.
Tech Lead
6

DevOps em modo validar, nao criar

DevOps nao roda scaffold completo. Em vez disso:

  1. Confere que as ferramentas listadas em DECISIONS.md rodam (executa os scripts de test/build ja existentes).
  2. Instala apenas o gap identificado pelo snapshot — tipicamente Playwright + diretorios tests/e2e/specs/ e tests/e2e/screenshots/.
  3. Adiciona .gitattributes com merge=union para docs/DECISIONS.md e docs/PROGRESS.md (pre-requisito do /dev-team-run em paralelo).
  4. NAO toca em package.json, configs ou deps existentes alem do estritamente necessario para o gap.
Regra de ouro: qualquer modificacao em arquivo existente do repo deve ser registrada como DEC-DEVOPS-ADOPT-XXX com diff explicito. Sem essa decisao, DevOps nao escreve no arquivo — vai para blocked/.
DevOps
7

PO cria tarefas tipadas (FEATURE / EXTEND / REFACTOR)

Tarefas em board/todo/ seguem o mesmo formato do greenfield, com um campo extra no frontmatter:

  1. kind: feature — greenfield-like, codigo novo isolado.
  2. kind: extend — Context aponta para o arquivo legado a ser estendido.
  3. kind: refactor — Acceptance Criteria inclui "comportamento externo identico ao baseline (validado por re-run dos testes existentes)".
PO Agent
✓

Validacao de cobertura e handoff para /dev-team-run

PO valida que todas as features do PRD tem tarefa correspondente. A partir daqui o fluxo se reune ao greenfield: /dev-team-run roda o loop autonomo identico, com a unica diferenca de que os agentes Dev sabem ler kind: da tarefa para decidir se criam arquivo novo ou editam existente.

Ciclo de Vida da Tarefa

Identico ao greenfield: todo/ → in_progress/ → done/ → verified/, com ramo blocked/. Veja workflow.html para o diagrama detalhado.

Artefatos Novos

docs/repo-snapshot.md

Output do code-archaeologist. Estrutura proposta:

  • Stack detectada: linguagens, frameworks, versoes (lidas dos manifests).
  • Endpoints existentes: lista METHOD /path → file:line.
  • Telas existentes: rota → componente principal → file:line.
  • Infra de teste: runners instalados, presenca/ausencia de Playwright.
  • Scripts: dev, build, test, lint — com comandos exatos.
  • Infra: Dockerfile, docker-compose, CI configs.

E gerado uma vez no PASSO 0 e nao e atualizado durante /dev-team-run — e um snapshot, nao um indice vivo.

DEC-TL-ADOPT-XXX / DEC-DEVOPS-ADOPT-XXX

Convencao de prefixo para decisoes especificas do fluxo brownfield, registradas em DECISIONS.md:

  • DEC-TL-ADOPT-001: Stack adotada do repo — cada item da tabela Stack vira uma entrada.
  • DEC-DEVOPS-ADOPT-XXX — toda modificacao em arquivo existente precisa de uma destas, com diff explicito.

Permite filtrar facilmente o que veio do legado vs o que foi acrescentado pelo orquestrador.

Campo kind: no frontmatter da tarefa

Adicionado ao YAML existente das tarefas em board/{status}/{ID}.md:

  • kind: feature — novo, isolado.
  • kind: extend — estende arquivo existente.
  • kind: refactor — reimplementa preservando interface.

Greenfield continua funcionando sem o campo (default implicito: feature).

Riscos & Salvaguardas

Sobrescrita de codigo legado

Mitigacao: DevOps em modo "validar, nao criar". Qualquer edicao em arquivo existente exige DEC-DEVOPS-ADOPT-XXX. Recomendado rodar todo o fluxo em branch isolada.

Wireframes que retro-documentam o existente

Mitigacao: PO so cria wireframe para UI nova ou alterada. Wireframes para tela existente → rejeitados pelo Tech Lead na revisao.

Contratos extends: apontando para codigo que nao existe

Mitigacao: Tech Lead confere via Grep que o path:line referenciado existe antes de salvar o contrato. Pre-flight obrigatorio.

Quebra silenciosa de fluxos legados

Mitigacao: smoke test final do QA precisa cobrir tambem caminhos do PRD que nao foram tocados diretamente, mas dependem de codigo modificado por EXTEND/REFACTOR.

Conflito stack adotada vs requisito do PRD

Mitigacao: se um requisito exigir tecnologia ausente (ex: PRD pede WebSocket, repo nao tem), Tech Lead registra DEC-TL-ADOPT-XXX propondo a adicao e a tarefa correspondente vai como kind: extend com escopo de infra.

Snapshot desatualiza durante o run

Mitigacao: repo-snapshot.md e gerado uma unica vez no PASSO 0 e tratado como imutavel durante o /dev-team-run, exatamente como os contratos. Mudou? Aborta o run e re-roda /dev-team-brownfield.
